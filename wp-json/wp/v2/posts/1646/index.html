How can you select the technological frameworks you should use for something that you’re building? How can you scale something that you’re building? Today’s GSF Accelerator mentorship session focused on this issue. The contributing speakers were\u00a0Ram Singla (Zomato),\u00a0Gajinder Singh (PayU India) and\u00a0Amit Sharma (Khojguru)\u00a0and discussed on how a development cycle should look like.<\/p>\n

Research First<\/strong><\/p>\n

Find the core value propositions first \u2014 for example, whether the mobile payment app or the hardware POS device is the core or whether local content aggregation or sentiment analysis is the core.\u00a0And it isn’t constant. It changes, it evolves.<\/p>\n

You need to find all the options that are available to you for the technologies that you would be using, and rate them on factors like time to market, enterprise, availability and cost, roadmap, legacy code, prior experience and the learning curve.<\/p>\n

Development<\/strong>\u00a0<\/em>– “Ship soon, get feedback, iterate often.”<\/em><\/p>\n

The development cycle goes from requirement to development to testing to review and finally releasing it. Then you take feedback from the user, fix bugs, add enhancements and release back \u2014 it happens again and again.\u00a0Get it out, experiment fast, fail fast.\u00a0You should\u00a0simplify the application from the perspective of the user and time to market.\u00a0Also, don’t be\u00a0dependent on one\u00a0platform. Use\u00a0open source platforms if you’re well versed with them.<\/p>\n

Don’t overbuild<\/strong>\u00a0–\u00a0“Find the mojo \u2014 where the\u00a0sweet spot\u00a0lies.”<\/em><\/p>\n

You should focus on building what you need today, not years down the line. If you would be guessing, you’d only\u00a0increasing maintenance burden.<\/p>\n

Explode<\/strong>\u00a0– “Decouple as many things as you can.”<\/em><\/p>\n

When you’re going to grow huge, you’d explode. You would need to have independent components, API driven interactions, separate application logic, and even need to cache like crazy.<\/p>\n

Analyze<\/strong><\/p>\n

You should know the flow of your program and be able to pinpoint a leak \u2014 can you get rid of the leak? How?\u00a0Can throwing more hardware help\/delay the problem? You need to engineer the bottlenecks \u2014 know the exact scenarios and how they will evolve by looking at the current ecosystem \u2014 for instance choosing between fingerprint or QR code.<\/p>\n

Plus, you should always be logging data, even if you’re not using or analysing it. It would help you in the future \u2014 for instance to determine user’s preferences, etc.<\/p>\n

Conclusion<\/strong><\/p>\n

    \n
  • Don’t tend to solve long-term problems, have a 6-month roadmap.<\/li>\n
  • Do it fast, push it out, let the customers give feedback. Anticipate what they would want next.<\/li>\n
  • Keep is simple.<\/li>\n
  • Be frugal, don’t waste money.<\/li>\n
  • Scalability: throw more hardware at it if it helps.<\/li>\n
  • Run the company on the data you collect.<\/li>\n<\/ul>\n

    Don’t stop.\u00a0<\/em>Innovate.\u00a0<\/em>Again and again.<\/em><\/p>\n
